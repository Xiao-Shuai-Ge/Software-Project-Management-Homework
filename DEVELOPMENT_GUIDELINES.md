# 项目开发准则 (Project Development Guidelines)

本仓库作为“软件项目管理”课程实验的练习项目，旨在通过“教学工作量管理系统”的文档管理，学习和实践 Git 配置管理流程。

## 1. 分支管理策略

本项目采用符合实际开发习惯的分支管理流程：

- **`release` 分支**：线上/生产分支，存放最稳定的代码，对应线上环境。
- **`main` 分支**：主开发/测试分支，所有功能的集成与测试点，对应测试环境。
- **`feature/*` 分支**：特性开发分支，从 `main` 迁出，完成后直接合入 `main` 进行测试。
- **`hotfix/*` 分支**：紧急修复分支，从 `release` 迁出，修复后合入 `main` 进行测试，验证通过后再合入 `release`。

### 操作规范
1. 禁止在 `release` 分支上直接进行文档编辑或代码提交。
2. 每一个新任务或文档修改需创建对应的 `feature/` 分支。
3. 功能开发完成后，先合入 `main` 分支，在测试环境验证无误后再由负责人合入 `release`。
4. 合并分支前，必须确保本地已拉取最新远程代码并解决冲突。

## 2. 提交规范 (Commit Message)

为了保持提交历史的清晰，请遵循以下格式：

`type: description`

### 常见类型 (Type)
- `feat`: 新增功能或文档
- `fix`: 修复问题/错误
- `docs`: 仅修改文档
- `style`: 格式调整（不影响逻辑）
- `refactor`: 重构
- `chore`: 构建过程或辅助工具的变动

### 示例
- `docs: 完成教学工作量管理系统需求分析说明书初稿`
- `fix: 修正 README.md 中的排版错误`
- `feat: 添加开发准则说明文件`

---

## 3. 并发操作与版本控制

### 冲突处理
在多人协作（并发操作）时，若发生冲突：
1. 先 `git fetch` 和 `git pull` 获取最新状态。
2. 在本地手动解决冲突。
3. 标记冲突已解决并重新提交。

### 标签管理 (Tagging)
每一个里程碑（如完成一个阶段的文档）应打上版本标签：
- `v1.0-alpha`: 初始文档架构完成
- `v1.1`: 详细设计阶段完成

---

## 4. 协作流程

1. **拉取 (Pull)**: 开始工作前，确保本地仓库是最新的。
2. **创建分支 (Branch)**: 基于最新 `main` 分支创建功能分支。
3. **提交 (Commit)**: 按照规范提交修改。
4. **推送 (Push)**: 将本地分支推送到远程。
5. **拉取请求 (Pull Request)**: 发起 PR 申请合并到 `main`，由其他成员进行 Review。
6. **合并 (Merge)**: 确认无误后合并并删除临时分支。
